## 2. Database Tables and Views

Important tables:

| Table | Purpose |
|---|---|
| `tree_master` | Master list of farm trees. `tree_no` is stored as text and is the primary key. |
| `harvest_records` | One row per harvested tree record. Stores tree number, harvest date, bunch values, generated totals, source, and optional ODK submission ID. |
| `harvest_cycles` | Defines harvest cycle date ranges, sale value, and cycle status. |
| `odk_sync_log` | Intended future log table for ODK import success, skip, failure, and retry tracking. |

Important views:

| View | Purpose |
|---|---|
| `vw_cycle_summary` | Cycle-level totals, total sale, total nuts, and sale price per nut. |
| `vw_harvest_records_with_cycle` | Harvest records with derived cycle information based on harvest date. |
| `vw_tree_history` | Tree master joined with harvest history. |
| `vw_daily_summary` | Daily harvest totals. |
| `vw_current_harvest` | Current open harvest cycle summary. |
| `vw_missing_trees_current_cycle` | Active trees not yet harvested in the current open cycle. |
| `vw_top_producers` | Lifetime tree production ranking. |

Important existing database protections:

- `tree_master.tree_no` is the primary key.
- `harvest_records.harvest_record_id` is the primary key.
- `harvest_records.tree_no` references `tree_master(tree_no)`.
- `harvest_records.odk_submission_id` has a unique index when not null.
- `harvest_cycles` has a no-overlap date range exclusion rule.
- `total_nuts` and `total_bunches` are generated by the database from bunch values.

Important gap:

- There is no database unique constraint on `(tree_no, harvest_date)`.
- Duplicate tree/date prevention is currently handled by import validation scripts, not by the database itself.

## 4. Duplicate Prevention Rules

Primary technical duplicate key:

- `odk_submission_id`

Business duplicate check:

- `tree_no + harvest_date`

Rules:

- Store `tree_no` exactly as received.
- Treat `tree_no` as text.
- Do not convert `tree_no` to integer, numeric, decimal, or float.
- Preserve values such as `845.1`, `1004.1`, `***`, `JF-101`, and future text-style identifiers.
- Never import the same `odk_submission_id` twice.
- Never silently import a second record for the same `tree_no + harvest_date`.
- If a duplicate is detected, report it and skip it unless a future explicit admin correction workflow is created.

Recommended duplicate handling:

| Situation | Recommended Action |
|---|---|
| `odk_submission_id` already exists | Skip and record as already imported. |
| Same `tree_no + harvest_date` exists | Do not import automatically; report as business duplicate. |
| Same ODK submission has changed after import | Do not update silently; require future admin review process. |
| Missing `odk_submission_id` | Reject record. |

## 5. Validation Rules

Every ODK harvest submission must pass validation before import.

Required validation:

1. Tree number
   - Must be present.
   - Must be treated as text.
   - Must exist in `tree_master`.

2. Harvest date
   - Must be present.
   - Must be a valid date.
   - Must belong to exactly one harvest cycle.
   - Must not fall outside all cycle date ranges.

3. Bunch values
   - `bunch1_nuts`, `bunch2_nuts`, and `bunch3_nuts` must be integers or blank.
   - Values must not be negative.
   - Blank values should be stored as null.
   - Totals must be calculated by the database, not trusted from ODK.

4. Locked cycle handling
   - If the matching harvest cycle is `Locked`, do not import automatically.
   - Importing into a locked cycle should require explicit approval or a future override flag.

5. Duplicate ODK submission handling
   - If `odk_submission_id` already exists in `harvest_records` or `odk_sync_log`, skip and report.

6. Duplicate tree/date handling
   - If a record already exists for the same `tree_no + harvest_date`, skip and report.

7. Source value
   - Future ODK imports must use `source = 'ODK'`.

8. Error handling
   - Invalid records must be reported clearly.
   - A validation failure should not partially import a batch.
   - The first implementation should be conservative.

## 6. Recommended Implementation

Do not build a full automatic sync first.

Recommended first implementation:

- Command-line script only.
- Dry-run by default.
- `--apply` required for any write.
- `--limit` option for small test batches.
- Full validation report before apply.
- Transactional insert.
- No deletion.
- No modification of existing records.
- No direct ODK database access.

Suggested future script:

```text
/home/muthu/muthu-harvest-dashboard/scripts/sync_harvest_odk.py
```

Suggested command examples:

```bash
python scripts/sync_harvest_odk.py --limit 5
```

```bash
python scripts/sync_harvest_odk.py --limit 5 --apply
```

Recommended script behavior:

1. Authenticate to ODK Central API.
2. Fetch submissions from Project 1, Form `muthu_harvest_v3_2`.
3. Normalize ODK fields into harvest database fields.
4. Validate all rows.
5. Print a report:
   - submissions fetched
   - valid rows
   - invalid rows
   - duplicates by ODK submission ID
   - duplicates by tree/date
   - missing trees
   - dates without cycles
   - locked-cycle conflicts
   - total bunches
   - total nuts
6. If not `--apply`, stop after report.
7. If `--apply`, insert valid records inside one transaction.
8. Insert ODK sync status into `odk_sync_log`.
9. Roll back the transaction if unexpected error occurs.

Recommended database insert fields:

| Database Field | Source |
|---|---|
| `tree_no` | ODK tree number field, preserved as text |
| `harvest_date` | ODK harvest date field |
| `bunch1_nuts` | ODK bunch 1 nuts field |
| `bunch2_nuts` | ODK bunch 2 nuts field |
| `bunch3_nuts` | ODK bunch 3 nuts field |
| `remarks` | ODK remarks field, if available |
| `photo_path` | Future optional media handling |
| `voice_path` | Future optional media handling |
| `odk_submission_id` | ODK instance/submission ID |
| `source` | Always `ODK` |

## 9. Explicit Non-Goals for the First Sync Version

The first harvest ODK sync must not:

- Run automatically on a schedule.
- Delete records.
- Modify existing harvest records.
- Modify ODK Central.
- Read ODK Central internal PostgreSQL.
- Import media files automatically.
- Override locked cycles automatically.
- Guess missing tree numbers.
- Convert tree numbers into numeric values.
